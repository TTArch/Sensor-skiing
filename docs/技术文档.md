## 2. React Native 与 Expo 是什么

### 什么是 React Native

React Native 是 Facebook（Meta）开发的一个框架，它让你**用 JavaScript（和 React）来写手机 App**，同时生成的 App 是真正的原生界面，不是网页。

```
传统开发方式：
  Android App  → 用 Java/Kotlin 写
  iPhone App   → 用 Swift/Objective-C 写
  两个完全不一样，要写两套代码

React Native 方式：
  用 JavaScript/TypeScript 写一套代码
  同时生成 Android App 和 iPhone App
```

### 什么是 Expo

Expo 是一套**在 React Native 基础上又加了工具和库的集合**，让开发手机 App 更容易。

```
React Native = 造汽车的底层工厂
Expo         = 提供了轮子、方向盘、发动机等现成零件，
              你只需要组装它们，不用从零造轮子

好处：
  - 写代码更快（很多常用功能已经做好了）
  - 不需要 Mac 电脑也能开发 iPhone App（云端构建）
  - 内置了很多常用工具：蓝牙、摄像头、语音、存储等
```

**我们的 App 用 Expo 构建**，相关配置文件是 `app.json`（应用配置）和 `package.json`（依赖管理）。

## 3. TypeScript 是什么，为什么用它

### 什么是 TypeScript

TypeScript（简称 TS）是 JavaScript（简称 JS）的"升级版"。它在 JS 的基础上加了一个功能：**类型检查**。

```javascript
// JavaScript（不标注类型）
function add(a, b) {
  return a + b;
}
add("hello", 5); // 结果是 "hello5"，不是报错！
```

```typescript
// TypeScript（标注了类型）
function add(a: number, b: number): number {
  return a + b;
}
add("hello", 5); // 写代码时直接报错：类型不匹配！
```

### 为什么这个项目用 TypeScript

```
好处1：提前发现错误     在写代码时就告诉你哪里写错了，不用运行 App
好处2：代码即文档        看函数参数类型就知道怎么用
好处3：IDE 自动补全      VSCode 知道每个变量是什么类型，补全更准确
```

我们项目里大量使用了 TypeScript 接口（interface），定义了所有数据结构的形状：

```typescript
// src/types/index.ts
// 定义"压力数据"长什么样
```

## 6. 传感器数据从哪里来

### 6.1 硬件层：压力传感器鞋垫

这是一种放置在滑雪靴内的设备，有 **5 个压力传感器点**：

```
    鞋垫俯视图（前为上）

    ┌─────────────────────┐
    │                     │
    │  [左前]   [右前]     │  ← frontLeft / frontRight
    │       [中间]        │  ← middle
    │  [左后]   [右后]    │  ← rearLeft / rearRight
    │                     │
    └─────────────────────┘

每个传感器返回 0-255 的原始值（电压/电阻转换为压力）
```

### 6.2 软件层：SensorAdapter 适配器模式

适配器模式（Adapter Pattern）是设计模式的一种：**把不同来源的数据转换成统一格式**，让后面的代码不用关心数据从哪来。

```
                 统一接口
            SensorAdapter 接口
                   │
      ┌────────────┼────────────┐
      │            │            │
      ▼            ▼            ▼
SimulatorAdapter  BleAdapter  未来可能有...
      │              │
      ▼              ▼
  模拟器数据       真实蓝牙数据
```

### 6.3 SimulatorAdapter 模拟器详解

模拟器是开发阶段用的，不用真实硬件就能测试。它的核心逻辑是**在几种滑雪状态之间平滑切换**：

```typescript
// src/utils/adapters/SimulatorAdapter.ts

// 定义了几种典型滑雪状态
const SKIING_STATES = [
  { name: 'flat_run',      // 平滑直滑
    pressure: { frontLeft: 52, frontRight: 52, middle: 45, rearLeft: 48, rearRight: 48 } },
  { name: 'back_seat',     // 后座滑（常见错误）
    pressure: { frontLeft: 30, frontRight: 30, middle: 40, rearLeft: 78, rearRight: 78 } },
  { name: 'forward',       // 过度前倾
    pressure: { frontLeft: 72, frontRight: 72, middle: 35, rearLeft: 28, rearRight: 28 } },
  { name: 'left_turn',     // 左转弯
    pressure: { frontLeft: 68, frontRight: 38, middle: 50, rearLeft: 62, rearRight: 42 } },
  // ... 更多状态
];

// 每次 tick：从当前状态插值到下一个状态，添加随机噪声
private tick() {
  // 计算当前插值进度（0到1）
  const progress = easeInOutCubic(stateElapsed / transitionDuration);

  // 在两个状态之间平滑过渡
  this.currentPressure = interpolate(stateA, stateB, progress);

  // 添加高斯噪声（模拟真实传感器的测量误差）
  const noisy = applyNoise(this.currentPressure);

  // 发射这一帧
  this.frameCallback({ pressure: noisy, timestamp: Date.now() });
}
```

每隔 500ms（0.5Hz）发送一帧。噪声使用 **Box-Muller 变换** 生成高斯分布随机数：

```typescript
private gaussian(): number {
  const u1 = Math.random();
  const u2 = Math.random();
  return Math.sqrt(-2 * Math.log(u1)) * Math.cos(2 * Math.PI * u2) * NOISE_VARIANCE;
}
```

## 7. 信号处理：把原始数据变成有用信息

`signalProcessor.ts` 负责把"原始传感器值"变成"干净的、有用的信息"，分三步：

```
原始值 (0-255)
    │
    ▼ normalize()         第1步：归一化
    │
原始值 (0-100)
    │
    ▼ applyLowPassFilter() 第2步：低通滤波（去噪声）
    │
平滑后的值
    │
    ▼ calculateCenterOfPressure() 第3步：计算压力中心点
    │
CoP { x: 0~1, y: 0~1 }
```

### 7.1 归一化（Normalize）

把 0-255 的原始值缩放到 0-100：

```typescript
function normalize(value: number): number {
  const clamped = Math.max(0, Math.min(100, value)); // 先限制在0-100范围
  return Math.round(clamped); // 四舍五入取整
}
```

### 7.2 低通滤波器（Low-Pass Filter）

传感器的原始数据会有噪声（比如抖一下突然显示95下一刻又显示90）。低通滤波器的作用是**让变化平滑**，不会因为单次抖动就大幅跳动。

本项目使用**指数移动平均（EMA）**：

```
filtered_value = previous_value + α × (new_value - previous_value)
α = 0.3（平滑系数，越大越敏感/不平滑）

举例：
上次值 = 50，这次新值 = 60
新滤波值 = 50 + 0.3 × (60 - 50) = 53
（下一次即使新值跳到90，滤波值也只会慢慢涨到54.1）
```

### 7.3 压力中心点（Center of Pressure，简称 CoP）

5个传感器 readings，我们想知道**总体的压力中心在哪里**。用加权平均：

```
            x方向（左右）
    0.0 ◄──────────────► 1.0
     内侧                  外侧

每个传感器的x坐标权重：
  frontLeft   = 0.2 （偏内侧）
  frontRight  = 0.8 （偏外侧）
  middle      = 0.5 （正中间）
  rearLeft    = 0.2
  rearRight   = 0.8

CoP.x = (p_FL×0.2 + p_FR×0.8 + p_M×0.5 + p_RL×0.2 + p_RR×0.8) / total_pressure
```

**CoP 有什么用？** 滑雪时，身体重心偏向哪边，就知道是在左转还是右转、转弯的深度如何。

## 8. 问题检测：规则引擎如何工作

`diagnostics.ts` 是核心算法模块。它用**阈值规则**检测6种常见滑雪问题。

### 8.1 什么是规则引擎

规则引擎就是一堆 `if-then` 规则。和机器学习的区别是：不需要训练数据，专家知识直接写成规则。

```python
# Python 等价逻辑
def detect_back_seat(pressure):
    front = pressure.front_left + pressure.front_right
    rear = pressure.rear_left + pressure.rear_right
    total = front + rear

    rear_fraction = rear / total if total > 0 else 0

    # 如果后脚压力占总压力超过60%，判定为后座滑
    if rear_fraction > 0.6:
        excess = rear_fraction / 0.6  # 超过阈值多少倍
        severity = 'high' if excess > 1.6 else 'medium' if excess > 1.2 else 'low'
        return Issue(type='back_seat', severity=severity)

    return None  # 没有问题
```

### 8.2 六种检测器一览

| 问题类型 | 检测逻辑 | 典型阈值 |
|---------|---------|---------|
| `back_seat`（后座滑） | rear / (front + rear) > 60% | rearFraction > 0.6 |
| `forward_pressure`（过度前倾） | front / (front + rear) > 60% | frontFraction > 0.6 |
| `upper_body_rotation`（上身转动） | 左右压力不对称 + 对角线不平衡 | lateralDeviation > 35% |
| `poor_angulation`（角度不够） | middle 压力过低且横向压力分布太平 | middle < 15 且 lateralSpan < 0.1 |
| `skidding`（过度刹车滑） | 总压力大 + CoP靠近中心 | total > 300 且 \|CoP.x - 0.5\| < 0.15 |
| `weight_imbalance`（重心偏移） | 左右脚压力比值过大 | heavy / light > 1.4 |

### 8.3 严重程度（Severity）怎么算

超过阈值越多，严重程度越高：

```python
def compute_severity(excess):
    if excess < 1.2:   return 'low'      # 轻微超标
    if excess < 1.6:   return 'medium'    # 中度超标
    return 'high'                        # 严重超标
```

### 8.4 帧级别 vs 趟级别检测

```
帧级别检测（每帧都跑）：
  detectBackSeat() → 这一帧有没有后座滑？

趟级别汇总（滑完才跑）：
  analysePressureHistory(所有帧)
      → 统计每种问题出现了多少次
      → 取每种问题的最高严重程度
      → 返回 aggregatedIssues[]
```

## 9. 数据存储：AsyncStorage

### 9.1 什么是 AsyncStorage

AsyncStorage 是 Expo 提供的一个**简单键值对数据库**，用来持久化存储数据。即使关掉 App，数据还在。

```
AsyncStorage = 一个存在手机里的持久化键值存储

存数据：  AsyncStorage.setItem('score', '85')
取数据：  AsyncStorage.getItem('score')  → '85'
存JSON：  AsyncStorage.setItem('runs', JSON.stringify([...]))
```

### 9.2 我们的存储内容

```typescript
// Zustand store 中配置的持久化字段
{
  pastRuns: [RunResult, RunResult, ...],  // 历史滑行记录（最多50条）
  isSimulated: true,                        // 是否模拟器模式
  language: 'en'                            // 语言偏好
}
```

### 9.3 存的是什么数据结构

```typescript
// RunResult = 一次滑行的结果（不含原始帧，节省空间）
interface RunResult {
  id: string;           // 唯一标识，如 "run_1710500000000"
  startTime: number;    // 开始时间戳（Unix毫秒）
  endTime: number;      // 结束时间戳
  duration: number;      // 持续秒数
  turnCount: number;     // 转弯次数
  score: number;         // 分数 0-100
  issues: SkiIssue[];    // 检测到的问题列表
  dominantIssue: IssueType | null;  // 最严重的问题
}
```

**为什么不用存原始帧数据？** 因为 RunResult 已经是分析后的摘要数据。原始帧数据（60帧×5个值）体积大、而且滑完就没用了，只用来做实时显示和诊断，不需要持久化。

### 9.4 Zustand 如何连接 AsyncStorage

```typescript
// src/store/index.ts
import { persist, createJSONStorage } from 'zustand/middleware';
import AsyncStorage from '@react-native-async-storage/async-storage';

export const useStore = create<AppState>()(
  persist(
    (set, get) => ({
      // 状态...
      pastRuns: [],
      addRunToHistory: (run) => {
        const { pastRuns } = get();
        const updated = [run, ...pastRuns].slice(0, 50); // 最多存50条
        set({ pastRuns: updated });
      },
    }),
    {
      name: 'ski-coach-storage',      // AsyncStorage 的键名
      storage: createJSONStorage(() => AsyncStorage),  // 用 AsyncStorage 持久化
    }
  )
);
```

## 10. 状态管理：Zustand

### 10.1 什么是状态管理

在 App 中，**状态**指的是随时会变的数据（比如当前连接状态、当前的分数、历史记录等）。状态管理就是**集中管理这些数据，让各个页面都能方便地读写**。

对比：Python 里的全局变量 + 单例模式，大约就是类似的概念。

### 10.2 Zustand 是什么

Zustand 是一个**超轻量的状态管理库**。相比 Redux，它的代码量少很多，但功能足够用。

```typescript
// 定义一个 store（类似一个类）
const useStore = create((set, get) => ({
  // 状态（类似类的属性）
  connectionStatus: 'disconnected',
  isSimulated: true,
  currentRun: null,
  pastRuns: [],

  // 方法（类似类的方法）
  setConnectionStatus: (status) => set({ connectionStatus: status }),
  addRunToHistory: (run) => {
    const { pastRuns } = get();  // 获取当前状态
    set({ pastRuns: [run, ...pastRuns].slice(0, 50) });
  },
}));
```

### 10.3 组件中怎么用

在任意组件里，可以这样读取和修改状态：

```typescript
// 在组件中读取状态
const { pastRuns, connectionStatus } = useStore();

// 在组件中修改状态（自动触发 UI 更新）
const { addRunToHistory } = useStore();
addRunToHistory(newRun);  // 调用后，HomeScreen 的历史列表自动更新
```

## 12. 语音播报：VoiceCoach

### 12.1 实现方式

使用 `expo-speech` 库，把文字转成语音播放：

```typescript
import * as Speech from 'expo-speech';

function speak(text: string) {
  Speech.speak(text, {
    language: 'en-US',    // 语言
    pitch: 1.0,           // 音调（正常）
    rate: 0.9,            // 语速（稍慢，方便听懂）
  });
}
```

### 12.2 useCoach 中的语音逻辑

```typescript
// useCoach hook 中的语音策略

// 规则1：只对 medium+ 严重程度的问题语音提醒
// 规则2：每种问题类型有 20 秒冷却时间
//         （同一问题重复说要有间隔，不然太烦人）

const COOLDOWN_MS = 20000; // 20秒

const onFrame = (frame: ProcessedFrame) => {
  const now = Date.now();
  for (const issue of frame.issues) {
    if (issue.severity === 'medium' || issue.severity === 'high') {
      const lastSpoken = lastSpokenRef.current[issue.type];
      if (now - lastSpoken >= COOLDOWN_MS) {
        lastSpokenRef.current[issue.type] = now;  // 更新冷却
        speak(issue.tip);                          // 语音播报
      }
    }
  }
};
```

## 13. 模拟器 vs 真实传感器

### 13.1 两种数据来源

```
数据来源A：SimulatorAdapter（模拟器）
  → 开发/演示用
  → 不需要真实硬件
  → 数据质量完美（只是加了点可控噪声）

数据来源B：BleSensorAdapter（蓝牙传感器）
  → 真实滑雪时用
  → 需要真实的压力传感器鞋垫硬件
  → 数据有真实环境的噪声和延迟
```

### 13.2 切换方式

```typescript
// 工厂函数：根据模式创建对应适配器
function createSensorAdapter(mode: 'simulated' | 'ble'): SensorAdapter {
  if (mode === 'simulated') {
    return new SimulatorAdapter();
  }
  return new BleSensorAdapter();
}
```

HomeScreen 上的"Demo Mode"开关控制 `isSimulated` 状态，决定用哪个适配器。

### 13.3 真实蓝牙传感器的接法（未来方向）

当要接真实硬件时，`BleSensorAdapter` 需要：

```typescript
async start(): Promise<void> {
  // 1. 请求蓝牙权限
  const permission = await Bluetooth.requestPermission();

  // 2. 扫描设备
  const device = await Bluetooth.scanForDevice(SERVICE_UUID);

  // 3. 连接设备
  await device.connect();

  // 4. 订阅数据通知（BLE Notify 机制）
  await device.startNotification(
    CHARACTERISTIC_UUID,
    (data: ArrayBuffer) => {
      // data = BLE 包，需要解析
      const pressure = parsePressurePayload(data);
      this.frameCallback({ pressure, timestamp: Date.now() });
    }
  );
}
```
